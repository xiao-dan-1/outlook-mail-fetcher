## 截图步骤与健康度

1. `01-home.png` — 初始等待输入状态。健康度：良好。信息架构完整，主要任务明确，但首屏信息量偏大，日志抽屉在桌面首屏中占据较强视觉权重。
2. `02-account-parsed.png` — 粘贴账号后自动解析。健康度：良好。解析反馈即时，隐私遮蔽有效，账号状态清楚；但“显示原文 / 隐私保护”文案切换需要更明确表达当前状态与动作结果。
3. `03-all-accounts-scope.png` — 切换为全部账号。健康度：良好。分段选择控件反馈明确；但选中范围与“选中账号 / 全部账号”的默认关系，对新用户仍需要更短解释。
4. `04-fetch-attempt.png` — 拉取后 OAuth 失败。健康度：中等。失败被及时展示，并同步到账号卡片、系统状态和日志；但日志错误直接展示长技术报文，用户难以判断下一步该改什么。
5. `05-log-expanded.png` — 日志展开。健康度：中等。抽屉展开后能看到更多细节；但它覆盖/压住邮件结果区上缘，和底部内容形成视觉竞争，像“浮层”和“页面内容”混在一起。
6. `06-dark-theme.png` — 深色主题。健康度：良好。整体色彩体系统一，核心文本可读；但错误日志中的长文本在深色底上更显拥挤，建议提供折叠详情。
7. `07-keyboard-focus.png` — 键盘焦点状态。健康度：中等。可聚焦控件数量合理，复制邮箱图标具备 aria-label；但小图标焦点目标的视觉反馈偏弱，部分禁用按钮只能靠 title 解释原因。
8. `08-mobile.png` — 390px 移动端。健康度：中等偏好。无横向滚动，模块能单列重排；但主 CTA 和结果区被大量设置项与日志挤下去，移动端完成任务需要长滚动。

## 强项

- 核心流程结构清楚：左侧账号管理，右侧控制台，下方邮件结果和日志，符合“先导入账号，再执行，再看结果”的任务模型。
- 输入后反馈很快：粘贴账号后立即显示 `1 / 1 行有效`、账号数和账号卡片，降低了用户对格式是否正确的疑虑。
- 隐私设计有意识：默认隐藏密钥，账号输入被遮蔽；这对包含密码、client ID、refresh token 的工具非常关键。
- 状态同步较完整：失败后系统状态、账号卡片、日志区都能反映结果，用户不会只在一个角落寻找反馈。
- 响应式基础良好：移动端没有横向滚动，桌面/深色主题均能稳定渲染。

## UX 风险

1. 错误信息太技术化。OAuth 失败时日志展示完整 HTTP 400、AADSTS、Trace ID、Correlation ID 等原始文本。对操作者来说，最重要的是“为什么失败”和“下一步做什么”，现在这两点被技术细节淹没。

2. 主要行动和次要行动层级有些挤。拉取邮件很突出，但复制验证码、导出、重试、隐私模式在同一行并列；失败后真正有用的“重试失败账号”没有被提升到更强的位置。

3. 日志抽屉视觉权重过高。运行日志是辅助诊断信息，但在首屏和展开状态里都非常抢眼；对普通用户来说，邮件结果和失败修复建议应比日志更优先。

4. 空状态做得美观但占空间较大。邮件结果区的骨架和详情空状态占据大量空间，移动端尤其明显，导致“拉取邮件”后的主要反馈需要滚动查找。

5. 一些按钮禁用原因依赖 title。比如复制验证码、导出结果等按钮在无结果时不可用，视觉上能看出禁用，但原因主要在 title 中，触屏用户和键盘/读屏用户不一定能获得同等解释。

## 可访问性风险

1. 技术错误被放进账号按钮的 aria-label。失败账号按钮的可访问名称包含完整错误 JSON，这会让读屏体验非常冗长，也可能暴露不必要的 trace/correlation 信息。建议 aria-label 只保留摘要，如“user@example.com，拉取失败，OAuth 错误，耗时 1.1 秒”，详情放到可展开区域。

2. 日志抽屉按钮有 `aria-expanded`，但缺少明确 `aria-controls` 关联。建议把按钮与日志区域 ID 关联，帮助辅助技术理解展开/收起影响哪个区域。

3. 小图标按钮视觉焦点偏弱。复制邮箱图标具备 aria-label，这是好的；但 34px 的小目标在视觉焦点状态下不够显眼，建议增加更强的 focus ring 和至少 40px 触控区域。

4. 颜色不能作为唯一状态信号。成功/错误使用绿/红标识，同时有文字“成功/错误”，这一点较好；继续保持不要只依赖色点。

5. 移动端目标密度偏高。390px 宽度下没有横向滚动，但控制台、日志、结果连续堆叠，用户需要大量滚动才能完成“导入—拉取—查看结果”的闭环。

## 推荐优先级

### P1：把失败状态改成“摘要 + 修复建议 + 技术详情折叠”

当前：`token refresh failed with HTTP 400... AADSTS... Trace ID...` 直接铺开。

建议：
- 主错误：`OAuth 授权失败：刷新令牌不可用于当前租户。`
- 下一步：`请检查 client ID、租户、refresh token 是否来自同一账号/租户，然后重试。`
- 次要入口：`查看技术详情`，展开后再显示 trace/correlation。

### P1：失败后提升“重试失败账号”和账号修复入口

失败状态中，“重试失败账号”只是并列按钮之一。建议在账号卡片或系统状态下直接给出：
- `重试失败账号`
- `编辑该账号`
- `复制错误摘要`

### P2：降低日志默认权重，把邮件结果/账号状态放前

日志对排障有用，但默认占据太多首屏注意力。建议：
- 默认折叠或降低高度；
- 失败时只显示一行摘要；
- 详细日志放在展开抽屉或右侧诊断面板。

### P2：改善移动端任务顺序

移动端建议排序为：账号输入 → 拉取 CTA → 当前状态/失败摘要 → 邮件结果 → 日志。当前日志夹在控制台和邮件结果之间，打断了任务闭环。

### P2：为禁用按钮提供可见原因

禁用按钮旁或按钮下方提供短提示，例如：
- `暂无验证码可复制`
- `拉取成功后可导出`
- `失败后可重试`

### P3：优化切换控件文案

`显示原文` / `隐私保护`、`隐私模式` 的当前状态和点击后的动作容易混淆。建议用开关式文案：
- `敏感信息：已隐藏`
- `显示原文 10 秒`
- `验证码隐私：开启/关闭`
